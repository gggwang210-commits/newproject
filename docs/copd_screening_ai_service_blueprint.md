# COPD 폐기능검사 필요도 선별 AI 서비스 설계 보고서

작성일: 2026-06-24  
문서 목적: Django 기반 COPD 폐기능검사 필요도 선별 AI 웹서비스의 서비스 내용, UX/UI, AI 모델, 데이터, 기술 아키텍처, 안전성, 사업성, 발표 방어 전략을 개발 착수 가능한 수준으로 정리한다.

## 0. 판단 표기 기준

- FACT: 확인된 사실 또는 제공 문서·공식 출처에 근거한 내용
- INFERENCE: 확인된 사실을 바탕으로 한 합리적 추론
- ASSUMPTION: 현재 프로젝트 진행을 위해 둔 가정
- RECOMMENDATION: 실행 권고
- RISK: 실패·오해·규제·운영 리스크

## 1. Executive Summary

### 핵심 결론

RECOMMENDATION: 이 프로젝트는 “COPD 진단 AI”가 아니라 “일반 건강검진·문진 정보 기반 폐기능검사 우선순위 선별 보조 서비스”로 포지셔닝해야 한다.

RECOMMENDATION: MVP는 모바일 웹 중심으로 구현하고, Django 단일 백엔드에서 문진 입력, 검진 수치 입력, ML 추론, 결과 저장, Doctor 검토 대시보드를 처리하는 구조가 가장 현실적이다.

RECOMMENDATION: 모델은 Logistic Regression을 기준 모델로 두고, LightGBM 또는 XGBoost를 최종 후보로 비교한다. 발표·의료 안전성 관점에서는 Recall, ROC-AUC, Calibration, SHAP 설명가능성을 함께 제시한다.

RECOMMENDATION: 사용자에게 직접 “COPD입니다”라고 보여주지 않고, “폐기능검사 상담 권장”, “의료진 검토 필요”, “응급 증상 시 즉시 진료”로 표현한다.

### 핵심 근거

FACT: COPD는 기침, 가래, 호흡곤란, 천명, 피로감 등을 동반할 수 있고, 흡연·대기오염·직업성 노출 등이 주요 위험요인이다.

FACT: COPD 확진에는 폐기능검사, 특히 spirometry가 핵심이다. 따라서 본 서비스는 진단 대체가 아니라 검사 필요도 선별에 머물러야 한다.

FACT: 내부 기획 문서 기준으로 KNHANES 2024 데이터, HE_COPD1 타깃, 40세 이상 유효 판정 3,091명, COPD 의심 295명, 정상 2,796명이라는 분석 가설이 존재한다. 이 수치는 코드 재현 전까지 제출 문서에서는 “프로젝트 내부 분석 기준”으로 표기한다.

RISK: 건강정보, 문진, 검진 수치, 예측 결과는 민감도가 높다. 회원가입, 저장, Doctor 대시보드, 예약 연계까지 들어가면 개인정보·보안·의료기기성 검토가 필요하다.

## 2. Service Blueprint

### 2.1 서비스 한 줄 정의

일반 건강검진 수치와 간단한 호흡기 문진을 바탕으로 COPD 고위험 가능성을 추정하고, 폐기능검사 상담 필요성을 안내하는 모바일 우선 AI 선별 보조 서비스.

### 2.2 해결하려는 문제

FACT: COPD는 중년 이후 만성 기침, 가래, 호흡곤란 같은 증상이 서서히 진행되어 사용자가 노화나 흡연 후유증으로 오해하기 쉽다.

FACT: 폐기능검사는 COPD 확인에 중요하지만, 모든 사람이 즉시 검사받는 구조는 현실적으로 어렵다.

INFERENCE: 건강검진 수치와 문진 정보를 활용해 고위험군을 우선 선별하면 의료기관은 검사 우선순위를 정하고, 사용자는 검사 필요성을 더 쉽게 이해할 수 있다.

### 2.3 기존 방식의 한계

- 사용자가 증상을 심각하게 인식하지 못한다.
- 일반 건강검진 결과와 호흡기 증상 정보가 분리되어 있다.
- 폐기능검사 대상자 우선순위 선정이 체계화되어 있지 않다.
- 병원 방문 전 사용자가 자신의 위험요인을 이해하기 어렵다.
- 문진 결과가 의료진에게 구조화되어 전달되지 않는다.

### 2.4 서비스 차별성

- 건강검진 수치와 문진 데이터를 결합한다.
- 진단이 아니라 검사 권고 트리아지에 집중한다.
- Patient, Technician, Doctor 역할을 분리한다.
- Doctor가 최종 검사 권고를 확정하는 Human-in-the-Loop 구조를 둔다.
- 모바일 입력, QR 접속, 카카오톡 공유, 검진기관 찾기까지 고려한다.

### 2.5 핵심 가치 제안

Patient: “내 증상이 검사 상담이 필요한 수준인지 쉽게 확인한다.”

Doctor: “고위험군을 우선 정렬해 폐기능검사 상담 대상자를 빠르게 검토한다.”

검진센터·보건소: “건강검진 후 사후관리 대상자를 구조화한다.”

팀 프로젝트·포트폴리오: “Django, ML, 헬스케어 UX, 데이터 모델링, 의료 안전성을 모두 보여준다.”

### 2.6 예상 사용자

Primary User:

- 40세 이상 사용자
- 현재 흡연자 또는 과거 흡연자
- 만성 기침·가래·숨참 경험자
- 건강검진 결과지를 보유한 사용자

Secondary User:

- 의사
- 간호사·검진센터 실무자
- 보건소 만성질환 관리 담당자
- 건강검진센터 관리자

### 2.7 주요 사용 시나리오

1. 사용자가 QR 코드 또는 카카오톡 링크로 접속한다.
2. “이 서비스는 진단이 아니라 검사 상담 필요도 안내”라는 고지를 확인한다.
3. 생년월일, 성별, 키, 몸무게, 흡연력, 기침, 가래, 호흡곤란, 혈압, 혈액검사 수치를 입력한다.
4. 모델이 COPD 고위험 가능성을 계산한다.
5. 사용자는 “낮음/주의/상담 권장” 단계와 주요 영향 요인을 확인한다.
6. 고위험 사용자는 가까운 검진기관 또는 의료진 상담 안내를 받는다.
7. Doctor는 대시보드에서 고위험 순으로 확인하고 검사 권고 여부를 기록한다.

## 3. User Journey

### 3.1 Primary User 분석

#### COPD 고위험군

- 연령: 40세 이상 중심
- 디지털 활용: 카카오톡 링크, QR, 문자 링크 접근 가능성을 우선 가정
- 건강 문해력: COPD, 폐기능검사, FEV1/FVC 같은 용어 이해도 낮음
- 입력 가능 정보: 나이, 성별, 흡연 여부, 기침·가래 여부는 쉽게 입력 가능
- 입력 어려운 정보: 혈액검사 수치, 검사 항목명, 과거 질환명
- 이탈 가능 구간: 회원가입, 검진 수치 입력, 긴 문진, 불안감을 유발하는 결과 화면
- UX 리스크: “고위험”을 “확진”으로 오해할 수 있음

#### 흡연자

- 입력 장벽: 흡연량을 축소 보고할 가능성
- UX 전략: “정확한 판단을 위해 솔직하게 입력해주세요” 문구 사용
- 결과 전략: 비난형 문구 금지, 금연 지원 정보 제공

#### 40세 이상 사용자

- 모바일 전략: 큰 버튼, 큰 글자, 한 화면 한 질문
- 접근성 전략: 작은 회색 글씨 금지, 진행률 표시
- 이탈 방지: “3분 안에 완료”처럼 완료 가능성을 제시

#### 호흡기 증상 보유자

- 위험: 심한 호흡곤란 사용자가 서비스를 응급 대응 도구로 오해할 수 있음
- 안전장치: “가슴통증, 심한 호흡곤란, 청색증, 의식저하 시 즉시 119 또는 응급실” 문구 필요

### 3.2 Secondary User 분석

#### Doctor

- 필요 정보: 원본 문진, 검진 수치, 모델 확률, 주요 영향 변수, 이전 기록
- UI 요구: 고위험군 우선 정렬, 클릭 한 번으로 검사 권고/보류 기록
- 리스크: 모델 결과를 과신하거나 반대로 무시할 수 있음
- 안전장치: 모델은 보조 정보이며 최종 판단은 의료진이 기록하도록 설계

#### 보건소·검진센터

- 필요 정보: 대상자 수, 고위험군 비율, 미응답자, 추적관리 현황
- UI 요구: 집계 통계, CSV 내보내기, 문자 안내 이력
- 리스크: 개인정보 처리 목적 초과 사용

## 5. UX/UI Design

### 5.1 모바일 우선 화면 흐름

#### 1) Landing

목적: 사용자가 5초 안에 서비스 목적을 이해한다.

주요 UI:

- 상단: “폐기능검사 필요도 간편 확인”
- 보조문구: “진단이 아닌 검사 상담 필요도 안내 서비스”
- CTA: “3분 문진 시작”
- 보조 CTA: “COPD가 뭔가요?”

이탈 방지:

- 첫 화면에서 회원가입 요구 금지
- 문진 예상 시간 표시
- 개인정보 입력 전 서비스 목적 설명

접근성:

- CTA 버튼 높이 48px 이상
- 텍스트 대비 확보
- 주요 안내는 색상만으로 표현하지 않음

#### 2) 서비스 소개

목적: 의료 오해 방지.

주요 UI:

- 카드 3개: “진단 아님”, “검사 상담 안내”, “의료진 확인 필요”
- 체크박스: “위 내용을 이해했습니다”

CTA: “동의하고 시작”

#### 3) 문진 시작

목적: 입력 부담을 낮춘다.

주요 UI:

- 진행률 0%
- 질문 그룹 안내: 기본정보, 흡연, 증상, 검진수치
- 임시저장 안내

CTA: “첫 질문으로 이동”

#### 4) 질문 입력

목적: 정확한 데이터를 수집한다.

UI 패턴:

- 한 화면 한 질문
- 선택형 우선
- 숫자 입력은 단위 고정
- “모름” 선택지 제공
- 이전/다음 버튼 제공

예시 질문:

- 현재 나이
- 성별
- 키, 몸무게
- 평생 담배를 5갑 이상 피운 적이 있는지
- 현재 흡연 여부
- 하루 평균 흡연량
- 기침 여부
- 기침 지속 기간
- 가래 여부
- 아침 가래 여부
- 숨참 정도
- 혈압 수치
- 혈당, HbA1c, 콜레스테롤 등 선택 입력

#### 5) AI 분석

목적: 대기 시간을 신뢰 형성 시간으로 사용한다.

주요 UI:

- “입력값을 확인하고 있습니다”
- “진단이 아니라 폐기능검사 상담 필요도입니다”
- Skeleton loading 1~2초

#### 6) 결과 제공

목적: 불안감 없이 다음 행동을 제시한다.

주요 UI:

- 위험 단계: 낮음 / 주의 / 상담 권장
- 확률: 숫자 단독 노출보다 구간 중심
- 게이지 차트
- 주요 영향 요인 3개
- “다음 행동” 카드

CTA:

- “의료진 상담 안내 보기”
- “검진기관 찾기”
- “결과 저장”

금지 UI:

- “COPD 확진”
- “정상입니다”
- “검사 불필요”
- 공포를 유발하는 붉은색 과다 사용

#### 7) 병원 안내

목적: 고위험 사용자에게 현실적 행동 경로 제공.

주요 UI:

- 현재 위치 또는 주소 기준 기관 목록
- 전화 버튼
- 길찾기 버튼
- 상담 준비물 안내

#### 8) 결과 저장·재방문

목적: 추적관리 가능성 확보.

주요 UI:

- 결과 PDF 저장
- 카카오톡 공유
- 30일 후 재확인 알림
- Doctor 의견 확인

### 5.2 위험도 색상 정책

- 낮음: 초록 계열, “현재 입력 기준 상담 우선도 낮음”
- 주의: 노랑 계열, “생활습관 관리와 추적 권장”
- 상담 권장: 주황·빨강 계열, “폐기능검사 상담 권장”

RISK: 색상만으로 위험도를 전달하면 색각 이상 사용자에게 정보가 손실된다. 반드시 텍스트와 아이콘을 병기한다.

## 6. AI Model Strategy

### 6.1 모델 후보 비교

| 모델 | 강점 | 한계 |
|---|---|---|
| Logistic Regression | 해석 용이, 의료 베이스라인 적합 | 비선형 패턴 약함 |
| Random Forest | 비선형 처리, 변수 중요도 제공 | 확률 calibration 필요 |
| XGBoost | tabular 성능 강함, 불균형 대응 가능 | 튜닝·설명 필요 |
| LightGBM | 빠른 학습, 대용량 tabular 적합 | 소규모 데이터 과적합 주의 |
| CatBoost | 범주형 변수 처리 강점 | 팀 학습 비용 존재 |

### 6.2 평가 우선순위

RECOMMENDATION: 선별 목적에서는 Accuracy보다 Recall을 우선한다.

평가 순서:

1. Recall: 고위험군을 놓치지 않는 능력
2. ROC-AUC: 전체 분류 성능
3. PR-AUC: 양성 비율이 낮을 때 유용
4. F1-score: Precision과 Recall 균형
5. Calibration: 예측 확률이 실제 위험과 맞는지 확인
6. Specificity: 과도한 검사 권고 방지

### 6.3 최종 추천 모델

RECOMMENDATION: 최종 발표 구조는 “Logistic Regression baseline + LightGBM/XGBoost 후보 비교 + SHAP 설명 + Recall 중심 threshold 조정”으로 간다.

MVP 1차 추천:

- Baseline: Logistic Regression
- Candidate: LightGBM, XGBoost
- 최종 배포 후보: LightGBM 또는 XGBoost 중 검증 성능과 calibration이 좋은 모델
- 설명 도구: SHAP summary, individual top factors
- 저장 방식: joblib

### 6.4 모델 운영 원칙

- 폐기능검사 관련 직접 진단 변수는 입력 변수에서 제외한다.
- HE_COPD1은 학습 타깃으로만 사용한다.
- Test set은 마지막 1회만 사용한다.
- Stratified split을 사용한다.
- class_weight 또는 scale_pos_weight로 불균형에 대응한다.
- threshold는 Recall 중심으로 조정하되, 과검사 위험을 함께 보고한다.

## 7. Data Strategy

### 7.1 데이터 출처

FACT: 내부 문서 기준 원자료는 KNHANES 2024 hn24_all.sas7bdat이며, 6,997명, 798개 변수로 정리되어 있다.

FACT: 내부 문서 기준 타깃 변수는 HE_COPD1이며, 0은 정상, 1은 COPD 의심, 9는 판정불능, NaN은 비대상으로 정리되어 있다.

FACT: 내부 문서 기준 40세 이상 유효판정 0 또는 1 대상은 3,091명이고, COPD 의심 295명, 정상 2,796명으로 클래스 불균형이 존재한다.

RISK: 이 수치는 코드와 원자료로 재현 검증 전에는 외부 발표에서 확정값처럼 말하면 안 된다.

### 7.3 결측·비해당 코드 처리

RECOMMENDATION:

- 8, 9, 88, 888, 999 등 비해당·모름 코드는 NaN으로 변환한다.
- 모델별로 결측 처리 전략을 분리한다.
- Logistic Regression은 imputation 필요.
- LightGBM/XGBoost는 결측 허용 여부를 검토하되 학습·추론 전처리를 동일하게 고정한다.

### 7.4 데이터셋 분할

- 40세 이상 필터링
- HE_COPD1 in {0,1} 필터링
- Test 20% holdout
- Train 80% 내 Stratified 5-Fold CV
- 최종 Test set은 모델 선택 후 1회만 사용

## 9. Safety & Compliance

### 9.1 의료 안전성 판단

RISK: 환자에게 직접 결과를 제공하는 선별 AI는 표현 방식에 따라 의료기기 소프트웨어 또는 임상 의사결정지원 소프트웨어 이슈가 발생할 수 있다.

RISK: “COPD 위험도 72%”를 사용자에게 단독 노출하면 확진처럼 오해할 수 있다.

RECOMMENDATION: 결과 문구는 “폐기능검사 상담 권장도” 중심으로 바꾼다.

### 9.2 필수 Safety Guardrail 10개

1. 모든 화면 상단 또는 결과 화면에 “진단 아님” 고지 표시
2. 결과 문구에서 “COPD 확진”, “정상”, “검사 불필요” 금지
3. 고위험 결과에는 “의료진 상담 권장” CTA 제공
4. 응급 증상 안내 별도 제공
5. Doctor 최종 검토 단계 설계
6. 모델 결과와 주요 영향 요인을 함께 표시
7. 모델 버전, 학습 데이터 기준, 한계 표시
8. 결과 저장 전 명시적 동의 수집
9. 민감정보 최소 수집 및 암호화 저장
10. 예측 오류·민원·이상 사례 기록 체계 운영

## 10. Business Strategy

### 10.1 고객 가설

- 건강검진센터: 검진 후 고위험군 사후관리
- 보건소: 만성질환 조기 선별 캠페인

- 보험·웰니스 기업: 건강관리 서비스 확장
- 개인 사용자: 폐기능검사 상담 필요성 확인

### 10.2 시장 문제

INFERENCE: COPD는 고령화, 흡연력, 대기오염, 직업성 노출과 관련되어 조기 선별 니즈가 존재한다.

INFERENCE: 건강검진 이후 “어떤 추가검사를 받아야 하는지”를 사용자 친화적으로 안내하는 서비스는 병원·검진센터의 후속관리 업무를 줄일 가능성이 있다.

### 10.3 수익모델 후보

1. B2B SaaS: 검진센터·보건소 대상 월 구독
2. 병원용 대시보드: 고위험군 정렬 및 사후관리 기능 제공
3. 캠페인형 납품: 지자체 금연·호흡기 건강 캠페인
4. 프리미엄 리포트: 개인 건강관리 리포트
5. API 제공: 검진 시스템과 연동

### 10.4 MVP 수익화 우선순위

RECOMMENDATION: 초기에는 B2C 과금보다 B2B/B2G 시범사업이 적합하다.

이유:

- 의료적 신뢰 확보 필요
- 개인이 위험도 결과에 직접 비용을 지불할 가능성은 불확실
- 검진센터·보건소는 사후관리 문제를 명확히 보유
- 공모전·정부지원사업과 연결하기 쉽다

   - 공격 포인트: 의료 정확성

   - 보완: spirometry 안내 추가

   - 공격 포인트: 건강정보 처리

   - 보완: 개인정보 처리방침 초안 작성

## 13. Final Recommendation

### 13.1 가장 현실적인 서비스 구조

Patient가 모바일로 문진과 검진 수치를 입력하고, 모델은 폐기능검사 상담 필요도를 산출하며, Doctor가 최종 검사 권고 여부를 기록하는 2~3역할 기반 트리아지 서비스.

### 13.2 가장 현실적인 UX 구조

회원가입 없는 체험 모드와 로그인 기반 저장 모드를 분리한다. 첫 시연은 “QR 접속 → 3분 문진 → 결과 → 상담 권장 → Doctor 대시보드” 흐름으로 만든다.

### 13.3 가장 현실적인 AI 모델

Logistic Regression baseline을 반드시 만들고, LightGBM/XGBoost를 비교한다. 최종 선택은 Recall, ROC-AUC, PR-AUC, calibration, SHAP 설명 가능성을 기준으로 한다.

### 13.4 가장 현실적인 기술 스택

- Frontend: Django Template + HTML/CSS/JS
- Backend: Django
- Database: SQLite 개발, PostgreSQL 배포
- Model: scikit-learn/LightGBM/XGBoost + joblib
- Visualization: Chart.js
- Deployment: Render 또는 Railway

### 13.5 공모전·포트폴리오 기준 추천안

- “의료 안전성을 고려한 AI 선별 보조 서비스”로 강조
- “진단 AI” 표현 금지
- Patient/Doctor 대시보드 연결을 핵심 데모로 구성
- KNHANES 데이터 기반 모델링과 Django 배포를 동시에 보여줌
- 심사 Q&A에는 데이터 누출, 성능, 규제, 개인정보 대응을 앞세움

### 13.6 실제 사업화 기준 추천안

- 개인 사용자 직접 판매보다 검진센터·보건소·병원 대상 파일럿으로 시작
- 의료진 검토를 포함한 B2B/B2G 워크플로우 우선
- 임상 적용 전 외부검증, 개인정보 영향평가, 법무·RA 검토 필요
- OCR, 예약 연계, 지도 API는 2차 확장 기능으로 분리
